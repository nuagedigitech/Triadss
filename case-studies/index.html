CASE STUDY – INTEGRATION OF MRO SYSTEM WITH SAP – Offline
For a low cost Airline operator

Overview

The Customer uses SAP for Finance Application and the Maintenance, Repair and overhaul functions have been implemented using Aviation solution. The MRO application must integrate with SAP for exchange and update of information. The offline integration requires conversion of MRO transaction data into SAP IDOC format.

Business challenges

As part of Maintenance repair and overhaul, Inventory management system, Order management system there are lot of data processed by Aviation solution. These transactions need to be sent to SAP. All the master data like customer master and vendor master will be updated in both the systems.

  • The process is tedious and redundant in nature and lots of precious efforts are involved.
  • Huge amount of file movements and tracking is required.
  • The mailing option has restrictions like file size limitation and security issues.
  • The network interruption is also another factor which affects timely updation.

Solution

The interfaces were identified for the data flow from MRO to SAP. All the Finance data to be converted into SAP IDOC compatible format at a pre defined interval say once in a day and the flat file will be pushed into a intranet server for SAP to process the records.

  • This addresses the secure and offline file transfer in intranet.
  • Automatic file transfer at predetermined intervals.
  • No middleware is used.

Business benefits

  • FTP over intranet.
  • Eliminate the need for middleware administration.
  • Automated data flow.
  • FTP logs are maintained.
  • Secured and only authorised access to data
CASE STUDY- INTEGRATION OF MRO SYSTEM WITH SAP –ONLINE
For an ERP solution provider

Overview

Our Solution was developed to integrate MRO application with SAP for exchange and update of information over real time. The online integration requires conversion of MRO transaction data into SAP IDOC format, SAP data need to be uploaded into Aviation application and a middleware for secured queuing of data for integration.

Business challenges

As part of Maintenance repair and overhaul, Inventory management system, Order management system there are lot of data processed by Aviation solutions. These transactions need to be sent to SAP and for some transactions taking place at SAP end like customer master, vendor master data creation and updation the data need to be updated in the MRO system.

  • The process is tedious and redundant in nature and lots of precious efforts are involved.
  • Huge amount of file movements and tracking is required.
  • The mailing option has restrictions like file size limitation and security issues.
  • The network interruption is also another factor which affects timely updation.

Solution

The interfaces were identified for integration and classified as outbound (MRO- SAP) and inbound (SAP- MRO). For all outgoing interfaces the MRO system data has been converted into SAP IDOC compatible format and all incoming data has been converted to MRO format. MQ has been used for secure and online transfer of data through various interfaces for each touch points.

  • This addresses the secure and offline file transfer in intranet.
  • Asynchronous nature of MQ allows uninterrupted transfer of data.
  • This has been built on IBM WEBSPHERE MQ technology.

Business benefits

  • Web based data transfer interface.
  • Automated data flow and validation at transaction level and interface level.
  • Administration of error log.
  • Secured and only authorised access to data.
CASE STUDY – Migration of database platform from Microsoft SQL to Oracle
For a leading Airline operator

Overview

ERP product had been developed using Microsoft Sql server 2008. The product had a constraint that it cannot be deployed on other database platforms. The objective was to provide Database flexibility. This also enables an organization to standardize the IT platforms. Migrating the database involves analyzing the objects that need to be ported and code conversion and testing of objects to ensure same functionality and response in the ported system.

Business challenges

The application uses database platform specific objects like tables, views, indexes, triggers, functions & stored procedures. All these objects are to be analyzed and the number of lines code that need to be migrated and tested has to be estimated.

Solution

Our Solution was to migrant the Objects from SQL to ORACLE, The objects that needed to be migrated are identified and industry standard tools are used for converting TSQL to PLSQL. However the automatic conversion was not 100% ready to use, as there are many instances of code conversion and functionality failures. After automatic conversion, manual debugging and testing of objects was performed for full functionality.

Business benefits

  • Independent database platform
  • Using the latest features of the platform
  • Increase in application performance
  • Able to standardize the database platform as the application can be deployed on multiple platforms
